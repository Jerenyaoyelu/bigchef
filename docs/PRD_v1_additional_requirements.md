# BigChef V1 补充需求文档（登录、视频跳转、大模型接入）

## 1. 文档目的

在当前 V1 主流程完成基础上，补齐以下关键能力：
1. 登录流程：匿名游客 + 手机号登录（先预留能力）
2. 查看视频：去抖音查看（外部兜底）
3. 推荐菜谱与查找菜谱接入大模型能力

该文档用于指导后续 Cursor 按阶段实现，避免需求分散。

---

## 2. 总体原则

- V1 以“快速可用 + 可演进”为目标，不追求一次做完完整账号体系。
- 内部数据优先，外部平台兜底（去抖音查看）。
- 大模型能力作为增强层，不阻断主流程可用性。

---

## 3. 需求一：登录流程（匿名游客 + 手机号登录预留）

## 3.1 目标
- 用户可在未登录状态下体验核心查询流程（游客模式）。
- 用户在需要保存个人数据（收藏、历史、家庭）时引导登录。
- 先预留短信登录接口与前端状态流，后续再接真实短信服务。

## 3.2 功能需求
- FR-LOGIN-01：应用启动默认游客可用。
- FR-LOGIN-02：收藏、浏览历史、社区发布等操作触发登录引导。
- FR-LOGIN-03：支持手机号验证码登录接口定义与前端调用链预留。
- FR-LOGIN-04：登录成功后游客状态可升级为用户态（迁移本地收藏/历史可选）。

## 3.3 接口预留（Auth）
- `POST /api/v1/auth/sms/send`
- `POST /api/v1/auth/sms/login`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/auth/me`

## 3.4 前端实现要求
- 增加 `AuthStore`（token、user、isGuest）。
- `http` 层支持 token 注入与 401 处理。
- 业务按钮支持“需登录”拦截器（如收藏）。

## 3.5 验收标准
- 未登录用户可完整使用推荐/查菜。
- 触发“需登录”动作时有明确引导，不出现静默失败。
- 预留接口与状态结构稳定，后续接短信服务时改动最小。

---

## 4. 需求二：查看视频（去抖音查看）

## 4.1 目标
- 当系统内部没有可展示视频时，用户可一键跳转抖音查看相关内容。
- 保证“有内容可看”的体验，不因内部冷启动造成断层。

## 4.2 功能需求
- FR-VIDEO-01：菜谱详情返回 `videoSourceType` 字段：
  - `internal`：内部视频可直接展示
  - `external_search`：无内部视频，展示去抖音入口
  - `none`：无视频
- FR-VIDEO-02：`external_search` 时返回 `videoSearchQuery`。
- FR-VIDEO-03：前端点击“去抖音查看”触发外部跳转（scheme/网页）。

## 4.3 返回结构建议
```json
{
  "dishId": "dish_101",
  "dishName": "可乐鸡翅",
  "videoSourceType": "external_search",
  "videoSearchQuery": "可乐鸡翅 家常做法",
  "videos": []
}
```

## 4.4 前端交互要求
- `internal`：显示“观看视频教程”
- `external_search`：显示“去抖音查看”
- `none`：显示“暂无视频”

## 4.5 验收标准
- 内部无视频时，页面不空白，用户可继续查看外部内容。
- 跳转行为可追踪（埋点：点击次数、来源页面、关键词）。

## 5. 需求三：推荐与查菜接入大模型能力

## 5.1 目标
- 提升推荐质量与查菜可用性，减少“查不到/结果弱相关”。
- 在不影响基础可用的前提下，引入模型增强。

## 5.2 场景拆分

### A. 推荐增强（按食材）
- 输入食材后，大模型参与：
  - 食材别名归一（番茄=西红柿）
  - 菜谱匹配重排（按可做性、耗时、口味偏好）
  - 缺失食材建议补齐

### B. 查菜增强（按菜名）
- 模糊菜名纠错（“可乐鸡腿” -> “可乐鸡翅”等候选）
- 未收录时返回“建议菜名候选”
- 后续版本可扩展为“AI 生成新菜谱并入库”（当前不实现）

## 5.3 功能需求
- FR-AI-01：推荐接口支持模型重排开关（可配置）。
- FR-AI-02：查菜接口支持候选菜名建议字段。
- FR-AI-03：模型异常时主流程可降级到规则/数据库结果。
- FR-AI-04：输出必须结构化，避免前端依赖自由文本。

## 5.4 推荐接口返回增强建议
```json
{
  "list": [...],
  "total": 10,
  "aiMeta": {
    "used": true,
    "model": "doubao-xxx",
    "reason": "rerank_by_cookability"
  }
}
```

## 5.5 查菜接口返回增强建议
```json
{
  "dishId": "dish_101",
  "dishName": "可乐鸡翅",
  "ingredients": {...},
  "stepsSummary": [...],
  "suggestions": ["红烧鸡翅", "啤酒鸡翅"]
}
```

## 5.6 验收标准
- 推荐结果相关性有明显提升（可用人工评估 + 指标跟踪）。
- 查菜失败时提供可点击候选，而不是直接终止。
- 模型不可用时仍可返回可用结果或空状态。

---

## 6. 埋点与指标（V1 补充）

- 登录相关：
  - `auth_prompt_shown`
  - `auth_sms_sent`
  - `auth_login_success`
- 视频相关：
  - `video_internal_click`
  - `video_external_douyin_click`
- AI 相关：
  - `ai_recommend_used`
  - `ai_suggestion_clicked`
  - `ai_fallback_used`

核心指标：
- 推荐页查询后点击率（CTR）
- 查菜命中率与二次搜索率
- 视频点击率（内部 vs 外部）
- 登录转化率（游客 -> 登录）

---

## 7. 风险与约束

- 短信服务会带来成本（当前先预留不接真实短信）。
- 外部平台数据不可控，V1 仅做跳转不做抓取。
- 大模型成本与稳定性需通过开关和限流控制。

---

## 8. 分阶段实施建议

### Phase A（本周）
- 补齐登录预留链路（游客态 + AuthStore + 需登录拦截）
- 完成 `videoSourceType + 去抖音查看` 前后端对接

### Phase B（下周）
- 推荐/查菜接入模型重排与建议输出
- 完善降级策略、埋点与监控

### Phase C（后续）
- 接入真实短信服务
- 准备“未收录 -> AI 生成并入库”能力（V1 不上线）
