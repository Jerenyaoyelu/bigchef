# AI 做饭助手 PRD v0.1

## 1. 文档信息

- 产品名称：AI 做饭助手（暂定）
- 版本：v0.1（MVP）
- 文档目标：指导 0-1 开发，优先上线可验证核心价值的最小可用版本
- 目标平台：微信小程序（优先），后续可复用到 App

## 2. 背景与目标

### 2.1 用户痛点

1. 家里有食材但不知道做什么，决策成本高。
2. 知道想吃什么，但要在多个平台反复搜索食材和做法。
3. 找到视频后步骤不清晰，新手执行门槛高。

### 2.2 产品目标（MVP）

围绕两个高频场景，做到“输入即得结果、结果可立即执行”：

1. 输入食材 -> 推荐可做菜 + 抖音做法视频列表。
2. 输入菜名 -> 返回食材清单 + 抖音做法视频列表。

### 2.3 核心指标（MVP 观察）

- 首屏结果返回时间 P95 < 3 秒
- 搜索到可执行结果比例 > 85%
- 首周用户次日留存 > 20%（早期验证口径）

## 3. 用户与场景

### 3.1 目标用户

- 新手做饭用户（18-35 岁）
- 下班后快速做饭人群
- 不愿长时间看图文教程的短视频用户

### 3.2 核心使用场景

1. “我有什么就做什么”：输入现有食材，快速确定今晚菜单。
2. “我想做这道菜”：输入菜名，一步拿到食材和视频教程。

## 4. 功能范围

## 4.1 本期范围（In Scope）

### A. 食材推荐（必做）

- 输入食材（文本）
- 菜品推荐列表（按匹配度排序）
- 展示缺少食材（可选补齐）
- 每道菜关联视频列表（抖音）

### B. 菜名查询（必做）

- 输入菜名（支持模糊匹配）
- 返回标准食材清单（主料/辅料/调味）
- 返回步骤摘要（AI 生成简化版，5-8 步）
- 展示相关抖音视频列表

### C. 结果页基础交互（必做）

- 排序：综合/最新/最热
- 筛选：预计耗时（例如 15 分钟内）
- 收藏菜品（本地或账号态，MVP 可先本地）

## 4.2 暂不做（Out of Scope）

- 拍照识别食材
- 购物下单
- 社区发布和评论
- 多平台视频聚合（先只接抖音）

## 5. 用户故事与验收标准

### US-01 食材推荐

作为一个下班后时间紧张的用户，我希望输入家里现有食材，快速得到可做菜和视频，以便马上开始做饭。

验收标准：

1. 输入 3-5 个食材，3 秒内返回候选菜列表。
2. 每道菜展示匹配度和缺少食材。
3. 每道菜至少返回 3 个可播放视频链接（或可跳转链接）。

### US-02 菜名查询

作为一个做饭新手，我希望输入菜名后直接看到食材清单与视频，避免反复搜索。

验收标准：

1. 输入常见菜名可命中标准菜谱。
2. 返回结构化食材：主料、辅料、调味。
3. 返回简化步骤（5-8 步，语言易懂）。

### US-03 快速决策

作为一个希望尽快吃上的用户，我希望结果里有耗时和难度，快速选一道最适合的菜。

验收标准：

1. 菜品卡片展示耗时、难度、口味标签。
2. 可按耗时筛选结果。

## 6. 信息架构与页面流程

## 6.1 页面结构

1. 首页（双入口）
   - 入口 A：输入食材
   - 入口 B：输入菜名
2. 结果页
   - 菜品列表 / 食材清单 + 视频列表
3. 菜品详情页
   - 食材、步骤、视频、收藏
4. 我的（可选，MVP 可简化）
   - 收藏列表

## 6.2 核心流程

### 流程 A：食材 -> 菜品

1. 用户输入食材（如：鸡蛋、西红柿、葱）
2. 服务端执行食材标准化（同义词归一）
3. 匹配引擎返回候选菜（含匹配度）
4. 视频服务并行拉取每道菜视频列表
5. 前端先渲染菜品卡片，再渐进加载视频

### 流程 B：菜名 -> 食材与视频

1. 用户输入菜名（如：可乐鸡翅）
2. 服务端菜名匹配（精确 + 模糊）
3. 返回标准食材清单 + 步骤摘要
4. 视频服务拉取关联视频并排序
5. 前端展示并支持一键开始做

## 7. 非功能需求

- 响应性能：核心查询 P95 < 3 秒
- 可用性：视频源异常时给出兜底文案和替代结果
- 可维护性：模块化服务设计，可替换视频源与 AI 服务
- 可扩展性：后续支持多平台视频、个性化推荐、拍照识别

## 8. 技术架构建议（先小后大）

采用“规则检索 + AI 增强”的分层架构，先确保稳定和可解释，再增强智能化。

### 8.1 模块划分

1. API Gateway
   - 统一鉴权、限流、日志
2. Recipe Service（菜谱服务）
   - 菜谱、食材、步骤、标签维护
3. Match Service（匹配服务）
   - 食材标准化、匹配度计算、排序
4. Video Service（视频服务）
   - 抖音检索、去重、打分排序
5. AI Service（AI 增强）
   - 文本理解、步骤摘要、解释文案
6. Data Layer
   - PostgreSQL（主数据）+ Redis（缓存）

### 8.2 架构原则

- 核心链路可解释：匹配分数来自明确规则与权重
- AI 非单点：AI 失败不影响基础结果返回
- 视频服务隔离：便于后续接入其他平台

## 10. 数据模型草案（MVP）

## 10.1 dishes（菜品表）

- id (pk)
- name
- alias_names（json）
- cuisine_type
- cook_time_minutes
- difficulty（1-3）
- taste_tags（json）
- created_at / updated_at

## 10.2 ingredients（食材表）

- id (pk)
- name
- alias_names（json）
- category
- created_at / updated_at

## 10.3 dish_ingredients（菜品食材关联）

- id (pk)
- dish_id (fk)
- ingredient_id (fk)
- role（main/secondary/seasoning）

## 10.4 videos（视频表）

- id (pk)
- dish_id (fk)

- source_video_id
- title
- url
- duration_sec
- like_count
- publish_time
- score
- fetched_at

## 11. 排序与推荐规则（MVP）

## 11.1 食材匹配分

- 基础分：命中主料数量 / 主料总数
- 惩罚项：缺失关键主料扣分
- 加分项：耗时短、难度低（新手友好）

建议公式（可配置）：

`match_score = 0.7 * main_hit_ratio + 0.2 * secondary_hit_ratio + 0.1 * ease_score - penalty`

## 11.2 视频排序分

- 质量信号：点赞、完播代理指标（可用时）
- 时效信号：发布时间衰减
- 内容信号：标题匹配菜名程度

## 12. 风险与应对

1. 抖音数据获取稳定性风险
   - 应对：视频服务增加缓存与兜底策略；失败时展示结构化步骤
2. 菜谱标准化复杂
   - 应对：先覆盖 Top 200 家常菜，逐步扩库
3. AI 结果不稳定
   - 应对：AI 仅做增强，不参与核心主链路决策

## 13. 研发计划建议（4 周）

- 第 1 周：数据模型 + 菜谱库 + 匹配引擎
- 第 2 周：视频服务接入 + 结果页联调
- 第 3 周：AI 文案/步骤摘要 + 兜底与缓存
- 第 4 周：小程序上线准备 + 灰度测试 + 指标看板

## 14. 验收清单（上线门槛）

1. 两条核心链路可用且稳定：
   - 输入食材 -> 推荐菜 + 视频
   - 输入菜名 -> 食材 + 视频
2. 接口响应性能达标（P95 < 3 秒）
3. 异常兜底完整（无白屏、无空结果死链）
4. 至少 50 道高频家常菜的数据质量验收通过
