# AI 做饭助手 技术方案 v0.2（App 导向）

## 1. 目标与约束

## 1.1 目标

在单人开发 + AI 辅助的条件下，3-4 个月内做出可持续迭代的 App 产品，覆盖：

1. 输入食材推荐菜品 + 视频
2. 输入菜名返回食材 + 视频
3. 拍照识别食材（v1.1 起）

## 1.2 现实约束

- 人力：你 1 人主导（产品 + 开发 + 运营），AI 负责提效
- 时间：有限，必须优先做高确定性功能
- 成本：早期控制在低固定成本、按量付费为主

## 2. 产品与技术总路线

采用“三阶段路线”，避免一开始追求全量原生导致成本过高。

### 阶段 A（0-6 周）：App MVP 可用

- 客户端：React Native + Expo（TypeScript）
- 后端：NestJS + PostgreSQL + Redis
- AI：LLM Provider 抽象层（先接豆包）
- 能力：双核心链路打通（文本输入类）

### 阶段 B（6-12 周）：增长与体验优化

- 加入登录、收藏、历史、个性化排序
- 视频卡片体验与加载策略优化
- 监控、埋点、A/B 对比框架

### 阶段 C（12 周后）：识别与原生增强

- 拍照识别食材（先云端推理，后评估端上推理）
- 对性能敏感模块做原生增强（相机/图像预处理/播放器）
- 保持主工程不重写，只做模块原生化

## 3. 技术选型（单人效率优先）

## 3.1 客户端：React Native + Expo（推荐）

推荐原因：

- 上手快、生态成熟、AI 生成代码质量高
- iOS/Android 一套代码，适合单人长期维护
- Expo 管理构建和能力接入，减少环境折腾

备选：Flutter

- 优点：UI 一致性强、性能好
- 不足：你当前 JS/TS 路线中断，AI 辅助与工程复用不如 RN 连贯

结论：优先 RN+Expo，后续必要时做原生模块扩展。

## 3.2 后端：NestJS（TypeScript）

推荐原因：

- 与前端同语言，认知切换小
- 模块化 + DTO + 校验 + 测试框架齐全
- 适合把 AI、视频聚合、推荐服务拆成清晰模块

## 3.3 数据与基础设施

- PostgreSQL：主业务数据（菜谱、食材、用户、收藏）
- Redis：缓存与限流（热点查询、视频结果缓存）
- 对象存储（后续）：图片上传与识别中间结果

## 3.4 AI 与识别

- 文本类 AI：豆包优先（保留多 Provider 抽象）
- 图片识别：
  - v1：云端视觉模型/API（最快）
  - v2：按成本与时延评估端上模型（可选）

## 4. 系统架构（可扩展）

核心原则：核心链路“可解释 + 可兜底”，AI 不做单点依赖。

模块划分：

1. API Gateway / BFF
2. Recipe Service（菜谱与食材知识）
3. Match Service（食材匹配与排序）
4. Video Service（抖音聚合与清洗）
5. AI Service（理解、摘要、解释）
6. Vision Service（拍照识别，阶段 C 引入）

调用关系：

- App -> API Gateway
- 文本查询：Gateway -> Match/Recipe/Video/AI -> 聚合返回
- 图片查询：Gateway -> Vision -> Match/Recipe/Video -> 聚合返回

## 5. 拍照识别功能设计（v1.1+）

## 5.1 用户流程

1. 用户拍照或从相册选择
2. 客户端做基础压缩与尺寸规范
3. 上传到识别服务，返回食材候选列表（含置信度）
4. 用户一键勾选修正
5. 进入“可做菜推荐”结果页

## 5.2 关键体验指标

- 拍照到结果：目标 2-4 秒
- Top-3 食材命中率：目标 > 80%
- 可编辑纠错：必须有（保证可用性）

## 5.3 技术要点

- 图片预处理：分辨率、亮度、旋转纠正
- 检测结果去重：如“西红柿/番茄”同义词归一
- 多食材框选：返回位置与标签（后续可做 UI 增强）
- 失败兜底：识别失败可切换文字输入

## 6. 数据模型新增建议（相对 v0.1）

新增表：

- `users`：用户账号基础信息
- `search_history`：查询历史（文本/图片）
- `favorites`：收藏
- `recognition_tasks`：识别任务记录（状态、耗时、结果）
- `feedback_logs`：用户纠错与结果反馈（用于迭代模型）

## 7. API 路线图（v0.2-v1.1）

v0.2（文本能力）：

- `POST /api/v1/recommend/by-ingredients`
- `POST /api/v1/dishes/search`
- `POST /api/v1/favorites`
- `GET /api/v1/favorites`

v1.1（识别能力）：

- `POST /api/v1/vision/recognize-ingredients`
- `POST /api/v1/recommend/by-recognition-task`

## 8. 单人 + AI 的研发工作流

## 8.1 代码协作策略

1. 你定义需求与验收标准（PRD + Issue）
2. AI 生成首版代码与测试
3. 你聚焦评审“业务正确性 + 用户体验”
4. AI 修复边界条件、补文档、补测试

## 8.2 每周节奏（建议）

- 周一：需求拆解与里程碑
- 周二到周四：功能实现
- 周五：回归测试 + 指标复盘 + 下周计划
- 每天固定 30-60 分钟清理技术债

## 8.3 AI 使用边界

- AI 负责：样板代码、接口联调、测试脚手架、文档同步
- 你主导：产品取舍、交互细节、数据口径、上线节奏

## 9. 里程碑计划（12 周版）

## 里程碑 M1（第 1-4 周）

- RN App 壳 + NestJS 后端可运行
- 两条文本核心链路完成
- 埋点与基础日志上线

验收：10-20 名种子用户可稳定使用。

## 里程碑 M2（第 5-8 周）

- 登录、收藏、历史、个性化排序
- 视频与结果页体验优化（首屏、骨架屏、失败兜底）

验收：7 日留存与核心转化有可观测提升。

## 里程碑 M3（第 9-12 周）

- 拍照识别食材 v1 上线（云端）
- 支持识别后纠错与一键推荐
- 收集识别反馈闭环

验收：识别链路可用，完成首轮精度与时延优化。

## 10. 成本预估（早期）

## 固定成本（每月）

- 数据库与 Redis：中低配云资源
- 日志与监控：基础版
- 对象存储：按量

## 浮动成本（按量）

- LLM 调用（文本理解与摘要）
- 视觉识别调用（拍照识别）
- 视频数据抓取与清洗任务（如使用代理服务）

策略：先限制免费用户调用频率，优先保证核心路径稳定。

## 11. 风险与应对

1. 风险：单人开发上下文切换过多
   - 应对：严格按里程碑冻结范围，不并行开太多功能
2. 风险：识别精度早期不稳定
   - 应对：强制用户可编辑纠错，形成训练反馈数据
3. 风险：第三方视频源波动
   - 应对：Video Service 独立，缓存 + 降级 + 多源预留
4. 风险：AI 成本不可控
   - 应对：缓存、短提示词、分层模型（高低配）
