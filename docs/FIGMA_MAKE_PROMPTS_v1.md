# BigChef V1 Figma Make Prompt 集合

本文档用于给 Figma Make 一次性输入设计要求，覆盖：
1. 需求一：登录流程（游客模式 + 手机号登录预留）
2. 需求二：查看视频（仅站内能力）
3. 需求三：推荐与查菜接入大模型（DB-first + AI fallback）

---

## Prompt 1：登录流程（游客模式 + 手机号登录预留）

```text
你是资深移动端产品设计师，请为 BigChef 设计「游客模式 + 登录引导 + 登录后数据升级」UI 方案，输出高保真移动端界面（iOS/Android 通用），并保持现有产品视觉一致。

【现有视觉风格约束】
- 底色 #fafafa，卡片白底，主色 #ff6b35，辅色 #4ecdc4
- 大圆角（12~24）、卡片化、轻插画/emoji 点缀
- 底部 Tab：推荐 / 查菜 / 我的
- 文案语言：简体中文，友好、清晰、少术语

【要设计的页面】
1) 登录引导弹层（由“收藏/历史/社区发布”触发）
2) 手机号验证码登录页（含发送验证码、倒计时、错误态）
3) 登录成功后的“数据升级中”过渡页
4) 升级结果反馈页（成功/失败可重试）
5) 我的页游客态 vs 登录态对比样式

【关键交互要求】
- 游客可以浏览推荐和查菜，但收藏、历史同步、发布等行为触发登录引导
- 登录成功后自动执行“游客数据迁移到账号”：
  - 状态1：迁移中（loading + 不可中断）
  - 状态2：迁移成功（显示迁移数量：收藏X条、历史X条）
  - 状态3：迁移失败（保留本地数据，支持“重试迁移”）
- 弹层中要明确价值点：登录后可跨设备同步收藏和历史
- 减少打断：支持“先逛逛（继续游客）”次级按钮（仅在非强制场景）

【组件与状态】
- 按钮：主按钮（橙色）、次按钮（描边/浅灰）
- 输入框：默认/聚焦/错误/禁用
- 验证码按钮：可点击、倒计时、重发
- Toast/Inline error：网络错误、验证码错误、迁移失败
- Loading skeleton 或进度提示：迁移阶段

【输出要求】
- 产出 5~7 张核心画板，覆盖完整闭环
- 每个页面标注关键状态与跳转关系
- 给出简要交互注释（触发条件、成功/失败分支）
- 保持与现有 BigChef 卡片风格统一，不要做成企业后台风
```

## Prompt 2：查看视频（仅站内能力）

```text
请为 BigChef 设计「菜谱详情视频模块」改版，目标是仅使用站内能力：有视频就播放入口，无视频就站内引导，不依赖外部平台跳转。

【视觉与产品基线】
- 延续现有 BigChef 风格：主色 #ff6b35，辅色 #4ecdc4，卡片化、圆角大
- 页面属于“查菜详情”场景，与现有“食材卡 + 步骤卡 + 收藏按钮”风格一致

【要设计的页面/状态】
1) 菜谱详情-有视频状态（videoSourceType=internal）
2) 菜谱详情-无视频状态（videoSourceType=none）
3) 点击“求更新该菜谱视频”后的成功反馈态（轻提示 + 按钮置灰/冷却态）

【无视频状态必须包含】
- 空状态视觉（图标 + 标题 + 说明）
- 两个行动按钮：
  - 主按钮：先看图文步骤
  - 次按钮：求更新该菜谱视频
- 反馈文案示例：
  - 提交成功：“已收到你的需求，我们会优先补充这个菜谱的视频”
  - 冷却提示：“你已提交过该需求，可稍后再来看看”

【交互规则要可视化】
- 同一用户对同一菜谱在冷却期内重复点击，不重复计数
- 点击“先看图文步骤”应定位到步骤模块（可用锚点反馈）
- 有视频时显示“观看视频教程”主入口；无视频时不出现播放按钮

【组件状态】
- 视频入口按钮：默认/按下/禁用
- 求更新按钮：默认/已提交（置灰）/冷却期
- 空状态卡：浅色背景 + 易读说明

【输出要求】
- 至少 3 张关键画板 + 1 张状态对照（internal vs none）
- 标注埋点建议位置：video_internal_click、video_empty_state_shown、video_request_update_click
- 保持单手操作友好，按钮触达区充足
```

## Prompt 3：推荐与查菜接入大模型（DB-first + AI fallback）

```text
请为 BigChef 设计「推荐与查菜的 AI 增强交互」，遵循 DB-first 策略：数据库命中优先，AI 仅在无结果或用户主动触发时使用。需要体现 AI 生成后入库复用，以及质量审核状态提示。

【现有体验基线】
- 推荐页：输入食材 -> 查询推荐 -> 卡片列表
- 查菜页：输入菜名 -> 详情结果或空状态
- 风格：暖色卡片化（主色 #ff6b35）

【要设计的页面与流程】
A. 推荐页（按食材）
1) DB命中结果页：显示常规推荐 + “AI帮我再推荐”按钮（非默认触发）
2) 用户点击 AI 按钮后的加载态
3) AI增强结果页：展示“AI生成/增强”标识、可解释信息（如缺失食材建议）

B. 查菜页（按菜名）
4) DB命中：直接显示菜谱详情（无AI打扰）
5) DB未命中：自动触发AI生成（loading）
6) AI生成成功：展示菜谱详情 + “已收录，后续可直接查询”提示
7) AI失败降级：友好空状态 + 重试/换关键词

C. 质量控制可视化（面向用户的轻提示）
8) AI菜谱状态标签设计：
   - 已校验（approved）
   - 复核中（needs_review，展示轻提示但不阻断）
   - 未通过（不展示给用户，后台态可忽略前台画面）

【交互与文案要求】
- 明确区分触发来源：
  - 自动触发（DB miss）
  - 手动触发（用户点 AI 按钮）
- 强调“先数据库、再AI”，避免用户误以为每次都走AI
- 对 AI 结果增加信任文案：
  - “该菜谱已通过自动结构校验”
  - “如有不准确，请以实际烹饪经验为准”

【组件要求】
- AI 标签（小徽标）：AI生成 / AI增强
- AI 按钮状态：默认、加载中、已完成
- 结果解释模块：触发原因、是否已入库、建议补充食材
- 失败态：网络异常/生成失败/内容校验失败

【输出要求】
- 8~10 张移动端高保真画板，覆盖推荐与查菜两条链路
- 每个关键页标注：触发条件、状态迁移、成功/失败分支
- 保持与现有 BigChef UI 风格一致，不做全新视觉体系
```
